# Guru Pipeline v3 — Design Doc

## Codename: Guru v3
## Scope: Benchmarking, smart sampling, fine-tuning track, and scoped optimizations for an evolving corpus

---

## 1. Summary

v2 produced a deployable artifact and locked the embedding model. v3 closes the next gap: the pipeline currently has no way to compare models, no principled way to sample chunks for evaluation or review, and no path to convert reviewed tags into a corpus-specific model. As the corpus grows from its current ~2K chunks toward 10K+ across new traditions, these gaps become operational pain — every model swap is a guess, every review session is unguided, and every new text starts the tagging clock from zero against an external model.

v3 adds four capabilities, in dependency order:

1. **Benchmark harness** — a reproducible way to compare any two tagging runs (model A vs. model B, prompt v1 vs. prompt v2, model vs. reviewed ground truth) with stored results and computed agreement metrics.
2. **Smart sampling** — stratified, embedding-diverse, and active-learning samplers that produce reproducible chunk sets for benchmarking and review prioritization.
3. **Fine-tuning track** — a deferred but pre-scoped pipeline for QLoRA fine-tuning a small open-weight model on accepted tags, producing a corpus-specific tagger that handles new texts cheaply.
4. **Scoped optimizations** — three pre-designed optimizations (embedding pre-filter, concurrent slots, tradition-scoped taxonomies) implemented only when the first complete run identifies the actual bottleneck.

Provenance changes to `staged_tags` are foundational and ship with pillar 1 so the rest of the work has clean data to consume.

**Estimated effort: ~5 days for pillars 1–2, ~3 days for pillar 3 (after first review pass), pillar 4 sized on demand.**

---

## 2. Goals and Non-Goals

### Goals

- Make every tagging run comparable to every other run via a stored, queryable benchmark record.
- Make sample selection reproducible: a sample set is a named, persisted artifact, not a one-off random draw.
- Track model and prompt provenance on every staged tag so future model swaps don't pollute the training set.
- Produce, when ready, a corpus-specific tagging model that can be self-hosted and runs ~3–4× faster than the 27B workhorse with comparable accuracy.
- Defer optimizations until measurement justifies them. No speculative complexity.

### Non-Goals

- Replacing the existing review workflow. `review_tags.py` and `review_edges.py` stay as the human-in-the-loop step.
- Changing the taxonomy or the prompt structure as part of v3 itself. The bench harness exists *to inform* those changes, not to make them.
- Online learning or continuous fine-tuning. Fine-tuning is a discrete, batch operation triggered manually after meaningful review accumulation.
- Multi-model serving. The pipeline runs one tagging model at a time; bench runs are sequential, not parallel across models.
- Cloud-hosted inference for steady-state operation. Fine-tuning may rent GPU time; tagging runs locally on the workstation.

---

## 3. Current State (v2)

### 3.1 What works

- Single SQLite store (`data/guru.db`) with embeddings, tags, and graph all colocated.
- `tag_concepts.py` with `--resume` and `tagging_progress` table — re-runs are safe and incremental.
- `staged_tags` with `status ∈ {pending, accepted, rejected, reassigned}` and a review CLI.
- llama.cpp + Qwen 3.5 27B Q4_K_M tagging on an RTX 3090 (24GB VRAM).

### 3.2 What's missing

- **No model provenance on tags.** `staged_tags` rows carry no record of which model produced them. Every row in the table is implicitly "whatever model ran last." This breaks the moment a second model is benchmarked against the first.
- **No sample reproducibility.** Any current "let me check 50 random chunks" is an ad-hoc draw with no record. Re-running a comparison on the "same" sample is impossible.
- **No agreement metrics.** Comparing two tagging passes today means eyeballing JSON diffs. There is no kappa, no per-concept agreement, no novelty rate, no latency record.
- **No path from accepted tags to a fine-tuned model.** The reviewed-tag dataset that *would* be ideal QLoRA training data is locked inside the staging table with no export tooling.
- **No scoped optimization plan.** Performance discussion is informal. The first complete run will produce data; v3 should have the optimizations pre-designed so implementation is mechanical when triggered.

## 4. Target State (v3)

### 4.1 Provenance, foundational

Two new columns on `staged_tags`:

```sql
ALTER TABLE staged_tags ADD COLUMN model TEXT;
ALTER TABLE staged_tags ADD COLUMN prompt_version TEXT;
```

Backfill existing rows with `(model='qwen3.5-27b', prompt_version='v1')` — the entire current `staged_tags` table was produced by a single model in a single prompt regime, so the backfill is a one-line `UPDATE`. Going forward, `tag_concepts.py` writes both fields on every insert.

A new top-level constant in `guru/prompt.py`:

```python
PROMPT_VERSION = "v1"  # bump whenever build_prompt() changes
```

This is the single thing that decides whether two runs are comparable. Any change to the prompt template, the score scale, the JSON schema, or the system prompt requires a version bump.

### 4.2 Bench harness

New file: `scripts/bench_tagging.py`. New tables in `scripts/schema.sql`:

```sql
CREATE TABLE IF NOT EXISTS bench_runs (
    run_id          INTEGER PRIMARY KEY AUTOINCREMENT,
    label           TEXT NOT NULL,           -- human label e.g. "qwen3-9b vs qwen3-27b round 1"
    model           TEXT NOT NULL,
    prompt_version  TEXT NOT NULL,
    sample_set_id   INTEGER NOT NULL REFERENCES sample_sets(set_id),
    pass_index      INTEGER NOT NULL DEFAULT 1,  -- which repetition of (model, sample_set, prompt_version)
    config_json     TEXT NOT NULL DEFAULT '{}',  -- temperature, max_tokens, seed, etc.
    started_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now')),
    completed_at    TEXT,
    UNIQUE (model, prompt_version, sample_set_id, pass_index, config_json)
);

CREATE TABLE IF NOT EXISTS bench_results (
    run_id          INTEGER NOT NULL REFERENCES bench_runs(run_id) ON DELETE CASCADE,
    chunk_id        TEXT NOT NULL REFERENCES nodes(id),
    tags_json       TEXT NOT NULL,           -- the parsed tag array as JSON
    raw_response    TEXT,                    -- optional: the raw LLM output for debugging
    latency_ms      INTEGER NOT NULL,
    error           TEXT,                    -- non-null if the call failed
    PRIMARY KEY (run_id, chunk_id)
);

CREATE INDEX IF NOT EXISTS idx_bench_results_chunk ON bench_results(chunk_id);
```

CLI shape:

```
bench_tagging.py run         --label LABEL --provider P --model M --sample-set ID
                             [--prompt-version V] [--passes N]
bench_tagging.py compare     --run-a A --run-b B [--output report.md]
bench_tagging.py compare     --run A --against-accepted [--output report.md]
bench_tagging.py determinism --model M --sample-set ID --passes N [--output report.md]
bench_tagging.py list
```

`run` executes a tagging pass against the sample set, writes to `bench_results`, and never touches `staged_tags`. Bench runs are deliberately isolated from the curation pipeline — they are measurements, not contributions. With `--passes N > 1`, the script executes N sequential passes, each persisted as its own `bench_runs` row with `pass_index = 1..N` sharing the same model/sample_set/prompt_version key. This enables determinism analysis without polluting the comparison surface.

`determinism` is a convenience that runs `--passes N` for one model and computes pairwise weighted-κ across all N passes, reporting the mean and variance of agreement. Useful for distinguishing "model A is genuinely better than model B" from "model A and B differ within their own sampling noise."

`compare` computes, per pair:

- **Presence agreement** per concept: of chunks where either model flagged concept C, what fraction did both flag?
- **Score agreement** (quadratic-weighted Cohen's κ on the {1, 2, 3} scale) for chunks where both flagged the same concept. See §4.2.1 for the choice of weighting.
- **Novelty rate**: percentage of chunks producing `is_new_concept=true` proposals.
- **Mean tags per chunk** and distribution.
- **Latency**: p50, p95, mean.
- **Per-tradition breakdown** of all of the above.
- **Top 10 worst-agreed concepts**, ranked by weighted κ ascending. With 57 concepts, the aggregate κ hides a lot — one concept at 0.2 can drag the average without being visible. Surfacing the bottom 10 directly in the markdown report makes the actionable failures obvious without making the report unreadable.
- **Full per-concept table** written to a sibling file at `docs/benchmarks/<run-a>-vs-<run-b>.concepts.json` for anyone who wants the complete matrix.

Output is markdown, dropped into `docs/benchmarks/<run-a>-vs-<run-b>.md`.

`compare --against-accepted` treats `staged_tags WHERE status='accepted'` as ground truth and computes the same metrics with the run as predictions. This is how fine-tune candidates get evaluated.

One special case when comparing against accepted: chunks where the model flagged a concept but the human marked every staged tag for that chunk as `rejected` are **hallucinated tags** — the model saw a concept that the human didn't. These get their own section in the report ("Hallucinated tags by concept"), counted separately from ordinary presence disagreement. They're not failures of kappa; they're the training signal for the next fine-tune. The rejected-tags-as-negative-examples path in §4.4's training export consumes exactly this category.

#### 4.2.1 Choice of agreement metric

The score scale is ordinal: 1 (peripherally present), 2 (clearly present), 3 (central theme). Two annotators disagreeing 1-vs-2 are closer to agreement than two disagreeing 1-vs-3. The metric should reflect that.

**Unweighted Cohen's κ** treats every disagreement as equally bad. A 1-vs-3 disagreement counts the same as 1-vs-2. *Pros:* simple to interpret, widely understood, no parameters. *Cons:* throws away ordinal structure; under-credits models that get the *direction* right but miss the *intensity* — exactly the failure mode small models exhibit on this task.

**Quadratic-weighted κ** penalizes disagreements by the square of their distance on the scale. 1-vs-2 contributes weight 1, 1-vs-3 contributes weight 4. *Pros:* matches the ordinal nature of the score scale; a 9B model that calls things "score 2" when the 27B says "score 3" is rewarded for being close, not punished as if it had said "not present"; standard in inter-rater reliability literature for ordinal scales (Fleiss-Cohen). *Cons:* slightly less intuitive — a κ of 0.7 doesn't mean "70% agreement," it means "70% of the way from chance to perfect, weighted by ordinal distance"; sensitive to the rare extreme-disagreement case (one 1-vs-3 weighs four times as much as one 1-vs-2).

**Linear-weighted κ** sits between the two: 1-vs-2 weight 1, 1-vs-3 weight 2. Smoother behavior than quadratic but doesn't punish extreme disagreement enough for our case — calling a "central theme" passage "peripherally present" is a meaningfully worse error than being one notch off, and we want the metric to say so.

**Decision: quadratic-weighted κ as the primary score-agreement metric.** Compute and report unweighted κ alongside it for sanity-checking and for readers who want the simpler number. If the two diverge sharply (e.g., unweighted κ = 0.4 but quadratic-weighted κ = 0.75), that's a signal the model is consistently off-by-one rather than randomly wrong, and worth surfacing in the report.

Presence agreement (the binary "did both flag this concept at all" check) stays as a separate metric — it answers a different question and isn't a κ at all.

### 4.3 Smart sampling

New file: `scripts/sample_chunks.py`. New table:

```sql
CREATE TABLE IF NOT EXISTS sample_sets (
    set_id          INTEGER PRIMARY KEY AUTOINCREMENT,
    label           TEXT NOT NULL,
    mode            TEXT NOT NULL CHECK(mode IN ('stratified','diversity','active')),
    params_json     TEXT NOT NULL,           -- mode-specific parameters
    chunk_ids_json  TEXT NOT NULL,           -- the actual sample as a JSON array of chunk ids
    seed            INTEGER NOT NULL,
    created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
```

Three modes:

**Stratified** (`--mode stratified --n 200 --floor 10`). Sample N chunks total, distributed across traditions proportional to corpus share, with a minimum of `floor` per tradition. Default mode. The baseline benchmark sample.

**Diversity** (`--mode diversity --n 200`). Run k-means with k=N on `chunk_embeddings.vector`, take the chunk closest to each centroid. Produces a sample that covers the conceptual range of the corpus rather than its statistical centre of mass. Best for catching model failure modes on rare passage types.

**Active** (`--mode active --n 100 --strategy disagreement`). Requires at least one prior `bench_runs` entry. Strategies:
- `disagreement`: chunks where two specified bench runs disagreed most (sum of absolute score differences across concepts).
- `low-confidence`: chunks where the most recent run produced only score=1 tags.
- `novelty`: chunks where the most recent run proposed `is_new_concept=true`.

Active samples feed review prioritization — these are the chunks where human judgment produces the most fine-tune signal per minute spent.

**Blind review.** When an active-sample set is loaded by `review_tags.py`, the reviewer sees the chunk and the candidate tags but *not* which model produced them or which model disagreed. This keeps judgment honest — a reviewer who knows "the big model said score 3 and the small model said score 1" will anchor on the big model. Disagreement provenance is recorded in the sample set's `params_json` and surfaced in the post-review bench report, but hidden from the review UI itself.

Every sample set is a row. Every bench run references a sample set. Reproducibility is automatic.

### 4.4 Fine-tuning track

Two scripts, both deferred until ≥1500 reviewed tags exist:

`scripts/export_training_data.py` — emits a JSONL file from `staged_tags WHERE status='accepted'`. Each line:

```json
{
  "system": "<frozen system prompt>",
  "user": "<the build_prompt() output for this chunk>",
  "assistant": "<canonical JSON tag array>",
  "chunk_id": "<for traceability, stripped before training>",
  "prompt_version": "v1"
}
```

Filters by `prompt_version` so training data is internally consistent. Hold-out is fully configurable via flags:

```
--holdout-tradition NAME     hold out one entire tradition (e.g. mesopotamian)
--holdout-traditions A,B,C   hold out several
--holdout-text TEXT_ID       hold out a single text by id
--holdout-fraction 0.1       random hold-out at the given fraction
--holdout-stratified 0.1     hold-out at given fraction, stratified by tradition
--no-holdout                 train on everything (only for production retrains after eval is settled)
```

The chosen hold-out spec is recorded in the model card so future fine-tunes can reproduce or deliberately rotate. Default for the first fine-tune: `--holdout-tradition mesopotamian` (smallest tradition, least training data lost). Subsequent fine-tunes should rotate the hold-out tradition or switch to `--holdout-stratified 0.1` once the per-tradition generalization signal has been collected at least once for each tradition.

Hold-out by tradition tests cross-tradition generalization (does the model learn the taxonomy or just per-tradition surface patterns?). Hold-out by random fraction tests within-distribution stability. Hold-out by text tests "can the model handle a new acquisition of the same kind it's seen before?" — this is the most operationally relevant test for steady-state corpus growth and probably the right default once you have enough reviews to spare.

Includes rejected tags as negative examples at a configurable ratio (default 20% of training mix), with score=0 entries — teaches the model what *not* to flag, which is half the value.

`scripts/run_finetune.sh` — wraps the existing Hermes QLoRA pipeline against the exported JSONL. Default base model: Qwen 3.5 8B. Default rank: 16. Trains, merges adapter, exports GGUF for llama.cpp consumption. Writes the resulting model card into `docs/models/<run-id>.md` with hyperparameters, training set hash, and eval-set bench results.

The fine-tuned model then runs through `bench_tagging.py compare --against-accepted` to validate before being adopted. No fine-tune is adopted as the production tagger without beating the base model on agreement metrics.

### 4.5 Scoped optimizations

Pre-designed, implemented on demand based on first-run data. Each gets its own short doc when implemented; below is the v3 commitment to the *shape*.

**Embedding pre-filter.** Embed each concept definition once at taxonomy load time (or persist them in `concept_embeddings`). For each chunk, compute cosine similarity against all 57 concept vectors and pass only the top K (default K=15) into `build_prompt()`. Drops prompt size ~4×, expected to drop wall-clock similarly. Triggered if the first run shows LLM call time as the dominant cost (it almost certainly will).

**Concurrent llama.cpp slots.** Start `llama-server` with `--parallel 4`. Run 4 tagging workers partitioned by `chunk_id` hash, all writing to the same `data/guru.db` (WAL mode handles this). Triggered if GPU utilization on the first run is below 80%.

**Tradition-scoped taxonomies.** Annotate each concept in `taxonomy.toml` with a list of applicable traditions. Filter the prompt accordingly. Triggered after enough review data exists to identify concept-tradition pairs that consistently produce score=0.

None of these are implemented as part of v3 itself. They are pre-designed escape hatches.

## 5. Migration steps

1. Schema migration: add `model` and `prompt_version` to `staged_tags`, backfill, add new tables. Single SQL file in `scripts/migrations/v3_001_provenance_and_bench.sql`.
2. `guru/prompt.py`: extract `build_prompt()` from `tag_concepts.py`, add `PROMPT_VERSION` constant.
3. `tag_concepts.py`: read `PROMPT_VERSION` from `guru/prompt.py`, write both provenance fields on every insert.
4. Implement `scripts/sample_chunks.py` with stratified mode first, then diversity, then active.
5. Implement `scripts/bench_tagging.py` with `run` first, then `compare`.
6. Self-bench: run Qwen 3.5 27B against itself on the same sample set — kappa should be ~1.0 modulo sampling temperature. This validates the harness.
7. Run first comparison: Qwen 3.5 27B (current) vs. Qwen 3.5 9B on a 200-chunk stratified sample. Write the report into `docs/benchmarks/`.
8. Defer fine-tuning track until ≥1500 reviewed tags exist. At that point, implement export and training scripts as a separate v3.5 minor.

## 6. Risks and mitigations

**Schema migration on a live DB during a long tagging run.** The `ALTER TABLE` calls are cheap but require an exclusive lock. Run them between the first complete pass and the first benchmark, not during.

**Bench results drift from real-world relevance.** If the bench sample is unrepresentative, the metrics are misleading. Mitigation: always include both stratified and diversity samples in any model evaluation, not just one.

**Training data leakage between fine-tune train and bench eval.** A chunk used in training that also appears in a future bench sample creates fake-good numbers. Mitigation: `bench_tagging.py` warns if a sample set overlaps any past training export.

**Fine-tune overfits on reviewer idiosyncrasies.** Small dataset, single reviewer. Mitigation: hold-out tradition as eval set, plus track per-reviewer if multiple reviewers ever exist.

**Optimization premature commitment.** Implementing all three optimizations before the first run produces complexity for unmeasured benefit. Mitigation: this doc explicitly defers them.

---

## 7. Success criteria

v3 is done when:

- Every row in `staged_tags` has model and prompt provenance.
- Two different models can be benchmarked against the same sample set with one command each, and compared with one more command, producing a markdown report.
- Sample sets are persisted and reproducible by ID.
- A first benchmark report (Qwen 3.5 27B vs. Qwen 3.5 9B) lives in `docs/benchmarks/`.
- The fine-tuning track is implementation-ready (scripts sketched, training format decided) even if not yet executed.
- The first complete run has informed which optimization, if any, is worth building.
