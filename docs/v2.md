# Guru Pipeline Migration — Design Doc

## Codename: Guru
## Scope: Migrate v1 Python pipeline to produce `guru-corpus.sql.gz`

---

## 1. Summary

The v1 Python pipeline was built as a local, exploratory curation environment. It stores the concept graph in SQLite and embeddings in ChromaDB, with a CLI-driven review loop (`tag_concepts.py`, `review_tags.py`, `propose_edges.py`, `review_edges.py`). It works, but its output format is not deployable — there is no single artifact that can be loaded into the production runtime.

This migration closes that gap by doing three things:

1. **Consolidate** — collapse ChromaDB into the existing SQLite database, so the pipeline has exactly one data store.
2. **Standardize** — switch to a fixed embedding model (Google `text-embedding-005`, 768 dimensions) and re-embed the corpus once.
3. **Export** — add `scripts/export.py`, a new script that emits `export/guru-corpus.sql.gz` — a single, idempotent, loadable SQL artifact that establishes the integration contract with `guru-web`.

Nothing else about the pipeline changes. `acquire.py`, `chunk.py`, `graph_bootstrap.py`, `tag_concepts.py`, `review_tags.py`, `propose_edges.py`, `review_edges.py`, `backfill_concepts.py` all remain as-is.

**Estimated effort: 2 days of focused work.**

---

## 2. Goals and Non-Goals

### Goals

- Produce a single `.sql.gz` artifact that `psql $DATABASE_URL` can load into a fresh or existing Postgres 17 + pgvector instance.
- Make the artifact idempotent: re-running the load command on an existing database fully replaces the corpus tables, leaving the app tables untouched.
- Make the export deterministic: running `export.py` twice on unchanged data produces byte-identical gzips.
- Remove ChromaDB as a dependency. The pipeline should have one data store, not two.
- Lock the embedding model to a named, versioned choice so vectors never drift.
- Make schema drift between `guru-pipeline` and `guru-web` loudly impossible.

### Non-Goals

- Rewriting the curation workflow. The CLI review tools stay.
- Adding incremental export. Full replace on every export is correct at this scale.
- Adding a web UI for curation. v1 is a local tool; that doesn't change.
- Supporting multiple embedding models simultaneously. One corpus, one model, at any given time.
- Real-time publishing. Corpus updates happen on a human cadence (weekly, monthly).

---

## 3. Current State (v1)

### 3.1 Data stores

- `data/guru.db` — SQLite. Contains `traditions`, `texts`, `chunks` (without embeddings), `concepts`, `edges`. This is the source of truth for graph structure.
- `data/vectordb/` — ChromaDB directory. Contains per-chunk embedding vectors, keyed by `chunk_id`, with concept metadata duplicated from SQLite via `backfill_concepts.py`.

The two stores are kept in sync by hand: chunk inserts go to SQLite, then `embed_corpus.py` populates ChromaDB, then `backfill_concepts.py` copies concept tags back into ChromaDB's metadata.

### 3.2 Problems this creates

- **Two sources of truth.** A chunk exists in both SQLite and ChromaDB, and it's possible (in practice, it happens) for them to disagree.
- **Two dependencies, two file formats, two backup targets.** ChromaDB is a substantial install footprint for what is functionally a `(chunk_id, float32[]) → ` lookup.
- **No export path.** Nothing today produces a single artifact that represents "the corpus at time T."
- **Embedding model is not recorded.** If the model is changed, existing vectors become silently incompatible with new queries.

### 3.3 What's working and stays

Everything upstream of embedding is solid and stays exactly as it is. The graph bootstrap, concept tagging, edge proposal, and review workflows are the actual intellectual work of the pipeline, and this migration does not touch them.

## 4. Target State (post-migration)

### 4.1 Data store

One file: `data/guru.db` (SQLite). A new table, `chunk_embeddings`, holds embedding vectors as `float32` blobs alongside the dimension and model name.

### 4.2 Embedding model

Google `text-embedding-005`. 768 dimensions, native. Chosen because:

- 768 is the native output, not a Matryoshka truncation — keeps the retrieval signal undistorted.
- ~$0.006 / 1M tokens — negligible cost for a <10K-chunk corpus.
- Competitive MTEB quality for English retrieval (not the best, but well above the bar for a RAG system at this scale).
- 2048-token context window is fine because chunks are already small (citation-addressable).

The model name is recorded in the `corpus_metadata` table on every export. If the model ever changes, the web app refuses to serve until a full re-embed has happened.

### 4.3 Export artifact

A single file: `export/guru-corpus.sql.gz`. Contents, in order:

1. `BEGIN;`
2. `DROP TABLE IF EXISTS ... CASCADE;` — all corpus tables.
3. The canonical `corpus-schema.sql` (CREATE EXTENSION, CREATE TABLE, no indexes yet).
4. `INSERT` statements for all rows, in foreign-key dependency order.
5. `CREATE INDEX` statements (especially the HNSW index, which benefits from being built after bulk load).
6. `INSERT` statements for `corpus_metadata` (last, so mid-load failures leave it unset).
7. `COMMIT;`

Wrapped in a single transaction so a failure mid-load leaves the pre-existing corpus intact.

## 5. Schema Changes to `data/guru.db`

Only one new table is added. Existing tables are untouched.

```sql
CREATE TABLE chunk_embeddings (
    chunk_id TEXT PRIMARY KEY REFERENCES chunks(id) ON DELETE CASCADE,
    dim      INTEGER NOT NULL,
    model    TEXT NOT NULL,
    vector   BLOB NOT NULL          -- float32 little-endian, `dim` values
);
```

Storage format rationale:

- `BLOB` with `float32` little-endian is the most compact portable format. A 768-dim vector is 3,072 bytes, versus ~15KB as JSON text.
- `dim` and `model` are stored per-row so a partial re-embed (e.g. model change mid-migration) is self-describing — each row knows what it is.
- `ON DELETE CASCADE` keeps embeddings in sync when chunks are removed during curation.

No changes to any other table. The concept-to-chunk metadata that `backfill_concepts.py` currently writes into ChromaDB is already in the SQLite `edges` table (`EXPRESSES` edges from chunks to concepts), so once ChromaDB is gone, `backfill_concepts.py` becomes a no-op and can be removed.

## 6. Migration Phases

### Phase 1 — Add `chunk_embeddings` table and backfill (half-day)

1. Write a one-shot migration script: `scripts/migrate_to_sqlite_embeddings.py`.
   - Opens `data/guru.db`.
   - Creates `chunk_embeddings` if absent.
   - Iterates ChromaDB; for each `(chunk_id, vector)`, inserts a row.
   - Prints count reconciliation: ChromaDB count vs. SQLite `chunks` count vs. `chunk_embeddings` count. Exits non-zero on mismatch.
2. Run it once locally. Confirm counts match.
3. Commit the new table to the schema (update `graph_bootstrap.py` if it owns schema creation, otherwise add a separate `schema.sql`).

At this point ChromaDB and `chunk_embeddings` are both populated and equivalent. Nothing else has changed.

### Phase 2 — Switch `embed_corpus.py` to write to SQLite (2 hours)

Modify `embed_corpus.py`:

```python
# Remove: chroma_client.add(...)
# Add:
import numpy as np
vec_bytes = np.asarray(vector, dtype=np.float32).tobytes()
conn.execute(
    "INSERT OR REPLACE INTO chunk_embeddings "
    "(chunk_id, dim, model, vector) VALUES (?, ?, ?, ?)",
    (chunk_id, len(vector), EMBEDDING_MODEL, vec_bytes)
)
conn.commit()
```

Also change the embedding model call itself to Google `text-embedding-005`. If the Google AI SDK is not already a dependency, add `google-generativeai` (or Vertex AI client, depending on auth preference).

Test: delete one chunk's embedding, re-run `embed_corpus.py` on that chunk, confirm the row is rewritten correctly.

### Phase 3 — Re-embed the full corpus (runtime varies, likely <1 hour)

Run `embed_corpus.py` with a flag to force re-embedding. For <10K chunks at Google 005's throughput, this should complete in minutes. Total cost: likely under $1.

Verify: sample a few chunks, confirm vectors are in `chunk_embeddings` with the correct `model` and `dim`.

### Phase 4 — Remove ChromaDB (15 minutes)

```bash
rm -rf data/vectordb/
pip uninstall chromadb
# Remove `chromadb` from requirements.txt / pyproject.toml
# Remove `backfill_concepts.py` — now redundant
```

Grep the codebase for `chroma` references and remove any lingering imports or config.

### Phase 5 — Add `corpus-schema.sql` to the repo (30 minutes)

Create `schema/corpus-schema.sql` using the canonical DDL. This file is checked into both `guru-pipeline` and `guru-web` and must be byte-identical between them. Add a CI job in both repos that fails if the hashes don't match.

### Phase 6 — Write `scripts/export.py` (1 day)

Structure:

```python
"""
Export the v1 SQLite corpus to a single .sql.gz artifact ready for
`gunzip -c ... | psql $DATABASE_URL`.
"""
from __future__ import annotations
import gzip
import hashlib
import sqlite3
import subprocess
from datetime import datetime, timezone
from pathlib import Path
import numpy as np

SCHEMA_FILE = Path("schema/corpus-schema.sql")
OUTPUT      = Path("export/guru-corpus.sql.gz")
SCHEMA_VERSION = 1
EMBEDDING_MODEL = "google/text-embedding-005"
EMBEDDING_DIM = 768


def git_sha() -> str:
    return subprocess.check_output(
        ["git", "rev-parse", "HEAD"]
    ).decode().strip()


def esc(s: str | None) -> str:
    if s is None:
        return "NULL"
    return "'" + s.replace("'", "''") + "'"


def esc_array(xs: list[str] | None) -> str:
    if not xs:
        return "NULL"
    inner = ",".join('"' + x.replace('"', '\\"') + '"' for x in xs)
    return f"'{{{inner}}}'"


def vec_to_pg(blob: bytes, expected_dim: int) -> str:
    arr = np.frombuffer(blob, dtype=np.float32)
    assert len(arr) == expected_dim, f"dim mismatch: {len(arr)} != {expected_dim}"
    # pgvector accepts text format: '[0.1,0.2,...]'::vector
    return "'[" + ",".join(f"{x:.7f}" for x in arr) + "]'"


def validate(conn: sqlite3.Connection) -> None:
    """Refuse to export if the corpus is inconsistent."""
    n_chunks     = conn.execute("SELECT COUNT(*) FROM chunks").fetchone()[0]
    n_embeddings = conn.execute("SELECT COUNT(*) FROM chunk_embeddings").fetchone()[0]
    if n_chunks != n_embeddings:
        raise SystemExit(
            f"Corpus inconsistent: {n_chunks} chunks vs "
            f"{n_embeddings} embeddings. Run embed_corpus.py."
        )
    # Verify all vectors are the expected dim and model
    bad = conn.execute(
        "SELECT COUNT(*) FROM chunk_embeddings "
        "WHERE dim != ? OR model != ?",
        (EMBEDDING_DIM, EMBEDDING_MODEL)
    ).fetchone()[0]
    if bad:
        raise SystemExit(
            f"{bad} embeddings do not match expected model "
            f"{EMBEDDING_MODEL} @ {EMBEDDING_DIM}d. "
            f"Re-run embed_corpus.py."
        )


def next_corpus_version(conn: sqlite3.Connection) -> int:
    """Monotonic counter persisted in a metadata table."""
    conn.execute("""
        CREATE TABLE IF NOT EXISTS _export_state (
            key TEXT PRIMARY KEY, value TEXT NOT NULL
        )
    """)
    row = conn.execute(
        "SELECT value FROM _export_state WHERE key = 'corpus_version'"
    ).fetchone()
    current = int(row[0]) if row else 0
    nxt = current + 1
    conn.execute(
        "INSERT OR REPLACE INTO _export_state (key, value) VALUES (?, ?)",
        ("corpus_version", str(nxt))
    )
    conn.commit()
    return nxt


def main() -> None:
    conn = sqlite3.connect("data/guru.db")
    conn.row_factory = sqlite3.Row

    validate(conn)
    version = next_corpus_version(conn)
    commit = git_sha()
    exported_at = datetime.now(timezone.utc).isoformat()

    OUTPUT.parent.mkdir(exist_ok=True)

    with gzip.open(OUTPUT, "wt", encoding="utf-8") as f:
        f.write("-- guru-corpus.sql.gz\n")
        f.write(f"-- Exported: {exported_at}\n")
        f.write(f"-- Source commit: {commit}\n")
        f.write(f"-- Corpus version: {version}\n\n")

        f.write("BEGIN;\n\n")

        # 1. Drop existing corpus tables
        f.write(
            "DROP TABLE IF EXISTS edges, concepts, chunks, texts, "
            "traditions, corpus_metadata CASCADE;\n\n"
        )

        # 2. Schema (from canonical file)
        f.write(SCHEMA_FILE.read_text())
        f.write("\n\n")

        # 3. Data, in FK order, deterministically sorted

        # traditions
        for r in conn.execute("SELECT * FROM traditions ORDER BY id"):
            f.write(
                f"INSERT INTO traditions (id, label, description, color) "
                f"VALUES ({esc(r['id'])}, {esc(r['label'])}, "
                f"{esc(r['description'])}, {esc(r['color'])});\n"
            )
        f.write("\n")

        # texts
        for r in conn.execute("SELECT * FROM texts ORDER BY id"):
            f.write(
                f"INSERT INTO texts (id, tradition, label, translator, "
                f"source_url, sections_format) VALUES "
                f"({esc(r['id'])}, {esc(r['tradition'])}, {esc(r['label'])}, "
                f"{esc(r['translator'])}, {esc(r['source_url'])}, "
                f"{esc(r['sections_format'])});\n"
            )
        f.write("\n")

        # concepts
        for r in conn.execute("SELECT * FROM concepts ORDER BY id"):
            f.write(
                f"INSERT INTO concepts (id, label, domain, definition) "
                f"VALUES ({esc(r['id'])}, {esc(r['label'])}, "
                f"{esc(r['domain'])}, {esc(r['definition'])});\n"
            )
        f.write("\n")

        # chunks (joined with embeddings)
        q = """
            SELECT c.*, e.vector, e.dim
            FROM chunks c
            JOIN chunk_embeddings e ON e.chunk_id = c.id
            ORDER BY c.id
        """
        for r in conn.execute(q):
            section_path = (
                r['section_path'].split('/') if r['section_path'] else None
            )
            f.write(
                f"INSERT INTO chunks (id, text_id, tradition, text_name, "
                f"section, section_path, translator, body, token_count, "
                f"embedding) VALUES ("
                f"{esc(r['id'])}, {esc(r['text_id'])}, {esc(r['tradition'])}, "
                f"{esc(r['text_name'])}, {esc(r['section'])}, "
                f"{esc_array(section_path)}, {esc(r['translator'])}, "
                f"{esc(r['body'])}, {r['token_count']}, "
                f"{vec_to_pg(r['vector'], EMBEDDING_DIM)}::vector);\n"
            )
        f.write("\n")

        # edges
        for r in conn.execute(
            "SELECT * FROM edges ORDER BY source, target, edge_type"
        ):
            f.write(
                f"INSERT INTO edges (source, target, edge_type, tier, "
                f"weight, annotation) VALUES ("
                f"{esc(r['source'])}, {esc(r['target'])}, "
                f"{esc(r['edge_type'])}, {esc(r['tier'])}, {r['weight']}, "
                f"{esc(r['annotation'])});\n"
            )
        f.write("\n")

        # 4. Metadata — last, so mid-load failures leave it unset
        f.write("INSERT INTO corpus_metadata (key, value) VALUES\n")
        f.write(f"  ('schema_version', '{SCHEMA_VERSION}'),\n")
        f.write(f"  ('embedding_model', '{EMBEDDING_MODEL}'),\n")
        f.write(f"  ('embedding_dim', '{EMBEDDING_DIM}'),\n")
        f.write(f"  ('corpus_version', '{version}'),\n")
        f.write(f"  ('exported_at', '{exported_at}'),\n")
        f.write(f"  ('source_commit_sha', '{commit}');\n\n")

        f.write("COMMIT;\n")

    # Print a summary so the operator sees what was exported
    size_mb = OUTPUT.stat().st_size / (1024 * 1024)
    print(f"Exported {OUTPUT} ({size_mb:.2f} MB)")
    print(f"  corpus_version:  {version}")
    print(f"  source_commit:   {commit[:12]}")
    print(f"  embedding_model: {EMBEDDING_MODEL}")


if __name__ == "__main__":
    main()
```

Key properties to preserve during implementation:

- **Single transaction.** Every `INSERT` is inside `BEGIN;`/`COMMIT;`.
- **Deterministic ordering.** Every query has an `ORDER BY` on primary key.
- **Metadata last.** If inserts fail, `corpus_metadata` stays unwritten, and the web app's schema-version check refuses to serve stale data.
- **Validation before write.** `validate()` refuses to export if embeddings are missing, model-mismatched, or dim-mismatched.

### Phase 7 — Round-trip test (2 hours)

```bash

SELECT key, value FROM corpus_metadata ORDER BY key;
SELECT tradition, COUNT(*) FROM chunks GROUP BY tradition ORDER BY tradition;
SELECT COUNT(*) FROM edges;
-- Self-similarity smoke test: a chunk's nearest neighbor is itself

    FROM chunks ORDER BY distance LIMIT 3;

# Tear down
docker rm -f guru-test
```

If all queries return sensible data — expected counts, self-similarity distance of 0 for the self match, sensible neighbors — the migration is complete.

## 7. Contract Between Repos

This migration establishes a single integration surface between `guru-pipeline` and `guru-web`: the loaded Postgres database, as defined by `schema/corpus-schema.sql`.

### 7.1 Version compatibility

Every export writes `schema_version` to `corpus_metadata`. The web app reads this at boot and refuses to start if it doesn't match its `EXPECTED_SCHEMA_VERSION` constant.

Procedure for a schema change:

1. Propose the change. Both repos must agree.
2. Update `schema/corpus-schema.sql` in both repos simultaneously (same PR semantics, same day).
3. Bump `SCHEMA_VERSION` in `export.py` and `EXPECTED_SCHEMA_VERSION` in the web app.
4. Ship web app first (new code, handles both old and new schemas if possible, or refuses old schema cleanly).
5. Run export. Deploy the new corpus.

### 7.2 Schema drift detection

CI in both repos verifies that `schema/corpus-schema.sql` matches a reference hash. The simplest implementation is a GitHub Actions job that fetches the file from the other repo on every push and compares hashes.

### 7.3 What's explicitly not in the contract

- The `data/guru.db` SQLite schema is an internal detail of `guru-pipeline`. The web app never reads it.
- The ChromaDB directory is gone; it was never part of the contract.
- Curation tooling (`tag_concepts.py` etc.) is internal. The web app doesn't care how chunks get tagged, only that tagged `EXPRESSES` edges appear in the loaded Postgres.

## 8. Operational Procedures

### 8.1 Publishing a corpus update

```bash
# 1. Curate as usual (add traditions, tag concepts, review edges, embed)

# 2. Export
cd guru-pipeline
python scripts/export.py
# → export/guru-corpus.sql.gz

# 3. Deploy
scp export/guru-corpus.sql.gz guru-vps:/tmp/
ssh guru-vps 'gunzip -c /tmp/guru-corpus.sql.gz | psql $DATABASE_URL'

# 4. Verify
ssh guru-vps 'psql $DATABASE_URL -c "SELECT * FROM corpus_metadata ORDER BY key"'
```

The TypeScript app picks up the new corpus on the next query. No redeploy.

### 8.2 Rolling back

Every export bumps `corpus_version`. To roll back, re-run the previous `guru-corpus.sql.gz` (which should be stored somewhere — at minimum, `git archive`d with a tag, or kept in object storage). The next load replaces the corpus in-place.

The daily `pg_dump` backup described in the v2 proposal covers the app tables, which are independent of the corpus. Corpus rollback and app-data rollback are separate concerns.

### 8.3 Failure modes

| Failure | Behavior | Recovery |
|---|---|---|
| `validate()` fails pre-export | Nothing is written | Fix the offending data in SQLite; re-run |
| `psql` fails mid-load | Transaction rolls back; old corpus intact | Investigate the SQL error; re-export if needed |
| `corpus_metadata.schema_version` mismatch at web boot | App refuses to serve | Either re-export with correct version, or roll back web app |
| Embedding model drift in SQLite (some rows wrong model) | `validate()` refuses to export | Run `embed_corpus.py` to re-embed offending rows |

## 9. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Re-embedding the full corpus corrupts graph associations | Re-embed is vector-only; `chunk_id` remains stable. Graph edges are in `edges` table, untouched by embedding. |
| `export.py` emits SQL that doesn't round-trip through `psql` | Phase 7 round-trip test against a real Postgres 17 container is mandatory before first production load. |
| Gzip artifact grows too large for manual scp | At 10K chunks × ~5KB/row including 3KB vector, raw SQL is ~50MB; gzipped, probably <15MB. Non-issue until 10x growth. |
| ChromaDB backfill script loses vectors due to ID mismatch | Count reconciliation at end of Phase 1 catches this. Re-embedding in Phase 3 is a belt-and-braces fallback. |
| Google API outage during embedding | Re-embedding is idempotent (`INSERT OR REPLACE`). Resume from the last completed chunk. |

---

## 10. Success Criteria

The migration is complete when all of the following hold:

- `data/vectordb/` no longer exists; `chromadb` is not in `requirements.txt`.
- `data/guru.db` contains a populated `chunk_embeddings` table with exactly one row per chunk, all with `model = 'google/text-embedding-005'` and `dim = 768`.
- `python scripts/export.py` produces a single `.sql.gz` file.
- That file, loaded into a fresh Postgres 17 + pgvector instance, produces a queryable database with correct row counts and working vector similarity search.
- Two consecutive exports of unchanged data produce byte-identical gzip files.
- `schema/corpus-schema.sql` exists and is byte-identical between `guru-pipeline` and `guru-web`, enforced by CI.
- The operational runbook in §8.1 has been executed end-to-end at least once against a real (non-production) Postgres.

---

## 11. Out of Scope

Explicitly not in this migration; may be addressed later:

- Any changes to curation CLIs or review workflows.
- A web UI for curation.
- Incremental export (delta-only updates).
- Supporting multiple embedding models simultaneously.
- Automated deploy from `export.py` to production Postgres. One manual `scp` + `psql` per release is correct for now.
- Feeding production query logs (`queries.chunks_used`) back into the pipeline for concept-graph improvement. The column shape is already pinned in `guru-web` for this future work, but no consumer exists yet in `guru-pipeline`.
